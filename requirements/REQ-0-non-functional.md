## REQ-0.1: Target Platform

**Status: âś… Implemented**

The server must run on Linux (any modern distribution) with Python 3.11+.

### Details
- Linux is the primary target platform (the server was designed and tested on Linux).
- No platform-specific code should be required; the server uses only cross-platform Python standard library modules plus `aiohttp` and `async_upnp_client`.
- Python 3.11 or later is required for `|` union type syntax and other modern language features.

---

## REQ-0.2: External Dependencies

**Status: âś… Implemented**

The server must minimize external dependencies. Only libraries that are strictly necessary for HTTP serving and UPnP should be used.

### Details
- **aiohttp >= 3.9.0**: HTTP server and HTTP client (for fetching remote streams).
- **async_upnp_client >= 0.47.0**: UPnP device framework, SSDP announcement/search responder.
- No other runtime dependencies.
- Build/packaging: `pyproject.toml` with standard Python packaging (no setuptools-specific features).

## REQ-0.3: LLM-Driven Development

**Status: âťŚ Not applicable (process requirement)**

The project follows a three-layer documentation/implementation approach:
1. **Requirements** (`requirements/`): High-level, user-facing behavior ("what the server must do"). Edited by the human.
2. **Specifications** (`specification/`): Derived from requirements by an LLM. Technical design detail sufficient for reimplementation ("how it works").
3. **Implementation** (`reciva_dlna_stream/`): Generated by a separate LLM instance from the specification, refined through tests.

This REQ-0.3 documents this process; it is itself not a feature to implement.

## REQ-0.4: Testing Framework

**Status: âś… Implemented**

The server must have an automated test suite covering all features. Tests must be integration-level where possible.

### Details
- Each test gets a fresh server instance (new port, new device UDN, new buffer).
- Test fixtures should use the same server startup path as production code (not mocks).
- Every feature must have at least one test.
- Tests must avoid timing-dependent behavior (use active queries like M-SEARCH instead of passive waiting for NOTIFY).
- Tests should verify: HTTP status codes, headers, body content, byte-level correctness for range requests.

## REQ-0.5: Error Robustness

**Status: âś… Implemented**

The server must be robust against expected failure scenarios.

### Details
- Remote stream connection failure (e.g. Icecast goes down): auto-retry with a delay (5 seconds).
- Client disconnect mid-stream: catch `ConnectionResetError`/`ConnectionAbortedError`, clean up task.
- Buffer timeout when waiting for data: return empty bytes, let the client decide what to do.
- Buffer offset trimmed (data unavailable because buffer is full and old data was discarded): raise error, log, close connection.
- All streaming tasks must be tracked and cleaned up reliably (no orphaned tasks).
- Unexpected exceptions must be logged with their full traceback for debugging.

## REQ-0.6: Logging Requirements

**Status: âś… Implemented**

The server must log meaningful information for debugging operational issues.

### Details
- Use Python `logging` with a module-level `_LOGGER`.
- Log levels:
  - `INFO`: Client connections/disconnections, major state changes.
  - `DEBUG`: Bytes forwarded, SSDP packet details, buffer state.
  - `WARNING`: Client disconnects mid-write, buffer timeouts.
  - `ERROR`: Unexpected exceptions (with `_LOGGER.exception()`).
- CLI `--verbose`/`-v` flag to enable DEBUG-level logging.
- **Per-chunk progress logging during streaming** (every chunk until 2 KB, then every 512 KB) is only emitted when `--verbose` is active. Without it, no periodic progress is logged during active streaming â€” reducing log noise in normal operation.

## REQ-0.7: Concurrency Model

**Status: âś… Implemented**

The server must be fully asynchronous â€” no synchronous blocking I/O.

### Details
- All I/O operations must be non-blocking (HTTP requests, stream reads, buffer management).
- Producer-consumer signaling must not lose wake-up events (no missed notifications between checking state and waiting).
- All shared mutable state must be properly synchronized to prevent data races.
- All streaming tasks must be tracked and cancellable on shutdown.
- Tight loops must not block the main execution context (they must yield periodically).
